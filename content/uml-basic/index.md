# UML 

**UML**
* 1997년 OMG (Object Management Group)에서 표준으로 채택한 통합 모델링 언어입니다
  * 그래디 부치, 이바 야콥슨, 제임스 럼바가 함께 UML을 개발하였습니다. 
  * 모델은 어떤 것을 실제로 만들 때 이렇게 만들면 잘 동작할지 미리 검증해 보는 것입니다.
  * 실제 물건을 만드는 비용보다 모델링 비용이 훨씬 적게 들 경우 모델을 만들어 설계를 검사합니다. 

**언급들**
  * 모델링의 목적은 주로 문서화하는 것이 아니라 이해하기 위한 것이다 - UML과 패턴의 적용
  * 스케치, 청사진, 프로그래밍 언어이다. 적어도 필자의 편경으로는, 이셋 중에서 가장 보편적인 것은 UML을 스케치로 생각하는 것이다. -UML Distilled
  * 적절한 상세 수준에서 UML 다이어그램을 작성하라 - Code Complete
  * 간혹 사람들이 UML 다이어그램과 같은 도구에 집착해서 그리기 쉬운 방향으로모델을 왜곡하곤한다. - 도메인 주도 설계
  * UML은 언어라는 것을 기억하라 - 객체지향 설계
 
**Stereotypes**
* UML에서 제공하는 기본 요소 외에새로운 모델 요소를 추가하기 위해 사용합니다.
* 기존에 정의 되어 있는 classifier에 스테레오타입을 추가하여새로운 모델 요소를 정의할 수 있습니다.
  * UML에서 추상화를 위한 모델링 엘러먼트를 classifier라고 총칭합니다. UML에서 가장 대표적인 classifier로는 유스케이스,클래스 등이 있으며 이외에 다양한 classifier들이 사용됩니다. 
* 예를 들어 네트워크를 모델링 할 때 <<라우터>>, <<스위치>>, <<허브>> 등에 대한 모델 요소를 추가하여 사용할 수 있습니다.

**Tagged Values**
* 모델 요소에 추가 정보를 추가하여, 모델 요소의 특성을 확장하는데 사용합니다.
* 속성 이름과 값을 쌍으로 설정할 수 있습니다.

**Constraints**
* 항상 지켜야 하는 내용이나 조건을 지정하기 위한 특성입니다.
* Note 요소를 사용하여 중괄호 {} 내에 제약 사항을 기술합니다.



# Usecase Diagram

**Usecase**
* 시스템이 무엇을 할지에 대한 기능 요구를 기술합니다.
* 시스템의사용자나 외부 시스템과의 상호작용을 정의하고 이해관계자들과의 의사 소통에 활용합니다.
* 누가 시스템을 어떻게 사용하는지, 시스템은 어떤 리소스들을 사용하는 기술합니다.
* 해당 기능이 내부적으로 어떻게 수행되는지는 기술하지 않습니다.
* Brief, Casual, Fully dressed의 세가지 형식이 존재합니다.
* 요구 사항을 수집 및 분석을 반복하며 Brief에서  Casual, Fully dressed 형식으로 발전 시켜 나값니다.
  * Breaf
    * 간결한 하나의 문장으로 요약하여 표현합니다.
    * 일반적으로 Main Success scenario 혹은 Happy path를 기술합니다.
  * Casual
    * 자유로운 여러 개의 단락 형식으로 요구 사항을 기술합니다.
    * Main success scenario와 Alternate Scenario들을 기술합니다.
    * 선택적으로 Exceptional scenario를 기술 할 수 있습니다.
  * Fully dressed
    * 모든 시나리오에 대해 use case에 표시 되어야 하는 모든 항목들을 기술합니다.
